1、任务概述

1.1、目标
（1）建设一套“校园失物招领系统”，面向校内用户提供寻物启事与失物招领的信息发布、查询、沟通与状态管理能力，提升失物找回效率。
（2）提供统一的移动端前端（uni-app）与后端服务（Node.js/Express），实现“注册/登录—发布—浏览—详情—留言—状态更新”的闭环流程。
（3）在发布场景中引入“AI 智能识别”能力：用户上传物品图片后，可一键生成用于失物招领信息的简洁描述，减少人工填写成本并提升信息规范性。
（4）与其他软件/模块关系说明：
- 本产品为一个完整的应用系统，包含前端 found/ 与后端 server/ 两部分。
- 后端对外提供 REST API（默认前缀 /api），并提供静态资源服务（/uploads）。
- AI 识别功能通过“兼容 OpenAI Chat Completions 的视觉接口”调用第三方模型服务（由环境变量配置）。

系统组成与接口关系（逻辑示意）：
前端（uni-app）
  ├─ 调用上传接口 /api/upload → 返回图片 URL（/uploads/xxx）
  ├─ 调用物品接口 /api/items → 发布/查询/详情/评论
  └─ 调用 AI 接口 /api/ai/recognize → 返回 description
后端（Express）
  ├─ MongoDB（用户/物品/评论数据）
  ├─ 文件系统 uploads/（图片文件）
  └─ 第三方 AI 服务（OpenAI 兼容接口，模型/端点可配）

1.2、系统（或用户）的特点
（1）用户群体：校内学生、教职工等移动端用户；少量管理员负责内容审核/处理（当前实现为管理后台“状态管理/删除”等基础能力）。
（2）使用频度预期：
- 日常浏览/搜索：高频（每次打开应用通常会加载列表）。
- 发布信息：中低频（丢失/捡到时使用）。
- 评论留言：中频（与发布者沟通）。
- AI 识别：中频（主要发生在发布时，取决于图片上传量）。
（3）系统特点（相较传统手工发布）：
- 支持图片上传与展示，单条信息最多 3 张图片；单张图片后端限制 5MB。
- 内置 AI 一键识别生成描述，识别结果可编辑。
- 以 JWT 为认证方式，受保护接口需登录。
- 管理员拥有管理后台页面，可按类型/状态筛选并更新状态、删除信息。

1.3、假定和约束
（1）运行假定：
- 后端运行在 Node.js 18+ 环境（使用原生 fetch）。
- 数据库使用 MongoDB，可通过 MONGODB_URI 配置。
- 前端通过 APP_CONFIG.api.baseUrl 指向后端地址（默认 http://localhost:3000）。
（2）安全与配置约束：
- 必须配置 JWT_SECRET，否则后端启动会直接退出（server/app.js 启动检查）。
- 使用 AI 识别功能需配置 OPENAI_API_KEY；AI 端点与模型由环境变量决定。
（3）功能约束：
- 图片上传仅允许 jpg/jpeg/png/gif/webp；文件大小不超过 5MB。
- 物品发布：标题与类型为必填；类型限定为 lost/found；图片数量 ≤ 3。
（4）非功能约束：
- 当前后端已实现基础错误处理中间件与 404 处理；尚未内置全局限流、审计日志、AI 调用配额控制。

2、需求规定

2.1、软件功能说明
2.1.1 用户与认证
- 注册（POST /api/auth/register）
  输入：username、password（必填），studentId/college/phone/wechat（可选）
  处理：参数校验、bcrypt 加密、保存用户（role 默认 user）
  输出：注册成功信息与用户基本信息
- 登录（POST /api/auth/login）
  输入：username、password
  处理：校验账号密码，生成 JWT（7d）
  输出：token 与 user 信息（含 role）
- 获取当前用户（GET /api/auth/me，需登录）

2.1.2 物品信息（失物/寻物）
- 发布信息（POST /api/items，需登录）
  输入：title、type（必填），category/description/location/time/images/contact（可选）
  处理：校验字段、创建 Item 记录
  输出：发布成功与 Item 数据
- 列表查询（GET /api/items）
  输入：type/category/keyword/status/page/pageSize（可选）
  处理：按条件过滤、分页、按创建时间倒序
  输出：{ data: list, total, page, pageSize }
- 详情（GET /api/items/:id）
  输出：Item 详情（包含 owner 部分字段）
- 修改（PUT /api/items/:id，需登录且为 owner）
  可更新：title/type/category/description/location/time/images/contact/status
- 删除（DELETE /api/items/:id，需登录且为 owner）
- 我的发布（GET /api/items/my，需登录）
  输出：当前用户的 items 列表（后端已实现）

2.1.3 评论留言
- 获取评论（GET /api/items/:id/comments）
  输出：评论列表（按时间正序，带 user 信息）
- 发表评论（POST /api/items/:id/comments，需登录）
  输入：content
  输出：新增评论（含 user 信息）
- 删除评论（DELETE /api/items/:id/comments/:commentId，需登录；作者或管理员）

2.1.4 图片上传
- 上传（POST /api/upload，需登录）
  输入：multipart/form-data，字段 file
  处理：multer 保存至 server/uploads，生成 url=/uploads/filename
  输出：{ url }
- 静态资源访问（GET /uploads/...）

2.1.5 管理后台（管理员）
- 获取信息列表（GET /api/admin/items，需管理员）
  输入：status/type（可选）
  输出：items 列表（带 owner.username）
- 更新信息状态（PUT /api/admin/items/:id/status，需管理员）
  输入：status=open|resolved
- 删除信息（DELETE /api/admin/items/:id，需管理员）
- 用户列表与角色更新（GET/PUT /api/admin/users...，已实现基础接口）

2.1.6 AI 智能识别（发布辅助）
- 识别接口（POST /api/ai/recognize，需登录）
  输入：{ imageUrl }
  处理：
  1) 将 /uploads/xxx 拼接为完整 URL（BASE_URL + imageUrl）
  2) 调用第三方“OpenAI 兼容 Chat Completions（含 image_url）”接口
  3) 提取 choices[0].message.content 作为 description
  输出：{ success: true, description }

2.2、对功能的一般性规定
（1）接口返回统一 JSON：成功返回 data/message/description 等字段；失败返回 message，并使用合适的 HTTP 状态码（400/401/403/404/500）。
（2）需要登录的接口必须携带 Authorization（token），后端解析 Bearer 或直接 token。
（3）前端统一通过 common/request.js 发起请求，失败由调用方决定 toast 文案。
（4）上传图片数量与大小限制需前后端一致：前端最多 3 张；后端校验 images 数量 ≤ 3，上传大小 ≤ 5MB。

2.3、用户界面
（1）前端为 uni-app 多端页面，整体风格简洁卡片化（浅色背景、卡片边框、圆角）。
（2）主要页面：
- 首页（列表、搜索、筛选：类型/状态）
- 发布页（表单 + 图片上传 + “AI 识别”按钮）
- 详情页（信息展示、图片预览、留言、标记已解决）
- 我的（登录态展示、我的发布、管理员入口）
- 管理后台（筛选、状态切换、删除）
（3）AI 交互：发布页在已上传图片时显示“🤖 AI识别”按钮；识别中显示 loading；成功后自动填入“详细描述”文本框。

2.4、对性能的一般性规定
2.4.1、精度
（1）数据字段精度：
- time 字段后端存 Date 类型；前端展示使用日期字符串（YYYY-MM-DD）。
- 文本字段（title/description/location/contact）以字符串存储，不做语义精度保证。
（2）AI 描述精度：
- AI 输出为“尽力而为”的文本描述，允许不确定性；最终以用户可编辑为准。

2.4.2、时间特性要求
（1）常规接口期望响应：
- 列表/详情/评论：一般 < 1s（取决于数据库与网络）。
- 图片上传：通常 0.5~2s（取决于图片大小与网络）。
- AI 识别：通常 2~5s（取决于第三方服务与网络）。
（2）前端已有加载/提示：列表加载有 loading；AI 识别有 showLoading。

2.4.3、灵活性
（1）可配置性：
- 后端通过 .env 配置 MONGODB_URI、JWT_SECRET、OPENAI_API_*、BASE_URL、PORT 等。
- AI 服务可切换不同兼容端点与模型（OPENAI_API_ENDPOINT/OPENAI_MODEL）。
（2）扩展性：
- Items/Comments/Users 通过 Mongoose Schema 管理，便于后续增加字段（如“拾到时间段/校区/物品细分类/置顶”等）。
- AI 提示词可在 server/routes/ai.js 调整。

2.4.4、输入输出要求
（1）输入类型：
- JSON：登录/注册/发布/评论/AI 识别等。
- multipart：图片上传。
（2）主要输入字段约束：
- username：3~20 位，字母数字下划线。
- password：≥ 6 位。
- phone：可选，需符合手机号格式（若提供）。
- item.title：2~100 字（后端校验）。
- item.type：lost|found。
- images：数组，≤ 3。
（3）输出类型：
- JSON：接口统一返回 JSON。
- 图片静态资源：/uploads/ 文件。

2.5、数据管理能力要求
（1）数据对象：
- User：用户名、密码哈希、学号、学院、电话、微信、角色。
- Item：标题、类型、分类、描述、地点、时间、图片数组、联系方式、owner、状态、创建/更新时间。
- Comment：关联 item、user、内容、创建/更新时间。
（2）规模估算（可按学校规模调整）：
- 用户：1k~50k；
- 信息：每日 10~500 条；
- 评论：每条信息 0~50 条。
（3）索引：
- Item：{ type, status, createdAt }、{ owner, createdAt }。
- Comment：{ item, createdAt }。

2.6、故障处理要求
（1）网络故障/后端不可用：前端提示“加载失败/请稍后重试”。
（2）认证失败：后端返回 401，提示“未提供 token / token 无效或已过期”；前端需引导用户重新登录。
（3）上传失败：
- 超过 5MB：返回“文件大小不能超过 5MB”。
- 非图片：返回“只允许上传图片文件”。
（4）AI 服务异常：
- 未配置 OPENAI_API_KEY：返回“AI服务未配置”。
- 第三方返回非 200：返回“AI识别失败，请稍后重试”。
（5）后端统一错误处理中间件：
- 处理 Mongoose ValidationError/CastError、JWT 错误等。

2.7、其他专门要求
（1）安全保密：
- API Key、JWT_SECRET 等敏感信息不得提交仓库；使用 .env 与部署环境变量管理。
- 建议生产环境启用 HTTPS、限制 CORS 源、加限流与审计日志。
（2）隐私：
- 用户上传图片可能包含个人信息（证件等）；需提示用户注意隐私遮挡。
- AI 识别会将图片 URL 提供给第三方服务，需在隐私政策中说明。
（3）可维护性：
- 路由按模块拆分（auth/items/upload/admin/ai）。
- 文档体系较完整（项目根目录多个 md）。
（4）可用性：
- 关键操作提供明确提示（发布中/识别中/加载中）。

3、运行环境规定

3.1、设备
（1）服务器（开发/测试）：
- CPU：2 核及以上（建议）
- 内存：2GB 及以上（建议）
- 磁盘：用于 uploads 图片存储，建议预留 ≥ 10GB（视规模）
（2）客户端：手机/小程序/H5（uni-app 支持的平台），能访问后端网络即可。

3.2、支撑软件
（1）后端：Node.js ≥ 18；npm。
（2）数据库：MongoDB 4.4+（建议）。
（3）依赖：express、mongoose、jsonwebtoken、bcryptjs、multer、cors、dotenv。
（4）前端：uni-app 运行环境（HBuilderX 或对应构建链）。

3.3、接口
（1）前后端接口：RESTful JSON，默认前缀 /api。
（2）认证协议：Authorization: Bearer <token>（或直接 token）。
（3）上传协议：multipart/form-data。
（4）外部 AI 接口：OpenAI 兼容 Chat Completions（messages 中包含 image_url），端点由 OPENAI_API_ENDPOINT 配置。

3.4、控制
（1）后端启动：
- 开发：server 目录下 npm run dev（nodemon）
- 生产：npm start
- 支持指定 ENV_FILE 读取不同 .env（如 npm run dev:local）
（2）关键控制信号来源：
- JWT_SECRET / MONGODB_URI / OPENAI_API_* / BASE_URL / PORT 等来自环境变量。
- 运行状态与错误通过控制台日志输出。

4、尚需解决的问题
（1）安全一致性：auth 中间件存在 JWT_SECRET 默认值（default_secret），但 app.js 启动强制要求 JWT_SECRET；建议统一策略并移除默认值以避免误用。
（2）“我的发布”前端实现：当前 found/pages/my/my.vue 通过 GET /api/items 拉取全量再本地过滤；后端已有 GET /api/items/my，建议前端改用该接口以减少开销。
（3）AI 配置与文档一致性：部分文档示例使用 OpenAI 官方端点/模型，而当前 ai.js 默认端点/模型为兼容服务（dashscope + qwen3-vl-plus）；建议在文档中明确默认实现或提供统一配置说明。
（4）生产级能力待补充：
- 频率限制（尤其是 /api/ai/recognize 与 /api/upload）
- AI 调用次数统计与成本监控
- uploads 存储策略（清理、对象存储、访问控制）
- 更完善的异常告警与日志（按请求追踪）
（5）合规与隐私：需要补充用户协议/隐私声明，明确图片与 AI 第三方处理范围。
